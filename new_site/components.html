Overview

Components in Sherpa are an abstraction of most of Twitter Bootsrap components through the use of Knockout JS. This was done to simplify coding of components and provide data integration to create dynamic prototypes.

In addition, Sherpa components can be customized once and used many times. When you update a component all instances within your prototype will change.

As you look at the components below you will single tags that render entire components. Behind the scenes the actual HTML exists but it is bound to data. Check out the raw component for dropdown buttons and view the source.

The anatomy of a Sherpa Component:

name:

Whatever the name of the component, the system will look for the directory "components/{name}". In the case where, "foobar" is the name of the component, Sherpa will set the component directory to components/foobar/.

type:

Type provides the specific name of the html file that renders the component. So let's say that in the reference to component "foobar" you also specify a type of "whatever". The system will get components/foobar/whatever.html. In the case where a type is not specified, the system will default the component name as type and look for the html in components/foobar/foobar.html

Note: Each component below may have various types that are standard to Sherpa in this current release. You may create your own types or complete components as you build your prototypes.

options:

Options provide further customization of each type of component. Each component will have a list of options available and which are defaulted when not specified.

When building your own component you can create as many option variables as you want to provide logic for the component to handle. It is up to you how complex you want to make a component but we suggest that when prototyping in agile environment you might want to opt for more "dumb" types of components to create variants then refactor as time goes to make your components smarter.

Page

A page component in Sherpa is not necessarely a whole page as you might think. It is rather a chunk of HTML that can be inserted into the screen. Why do we call this a "page" and not a module, component, view or something like that? These pages have the unique capability of being addressable as a page.

Explanation:

Let's say you have a page in your prototype: http://www.myserver.com/foobar.html. Inside this page you have tabs that refer to three sections, section1, section2, section3. Furthermore, inside section 3 you have another set of tabs with subsection1 and subsection2.

  http://www.myserver.com/
  ├── pages/
  │   ├── section1.html
  │   ├── section2.html
  │   ├── section3.html
  │   ├── subsection1.html
  │   ├── subsection2.html
  │
  foobar.html

A common requirement is to be able to address sub-section 2 directly (deep dive). This makes it a "page" and it is addressable by going to http://www.myserver.com/foobar.html#section3/subsection2

Markup

Rather than using the HTML provided in Bootstrap, Sherpa has custom code to better simplify and structure HTML. Paired with a data reference such as the one below, one custom tag takes care of all of the work.

 

< div data-bind="component: { 
	name :'dropdown', 
	type :'button',
	data : simple_dropdown_data,
	options:{
		color : 'primary',
		size : 'large'
	}
}"></div>

Options

You may have noticed the data-bind tag above with various elements.

name:

Whatever you put in the name, the system will look for the directory "components/{name}". In this case, "dropdown" will set the component directory to components/dropdown/.

type:

Type provides options for the trigger mechanism. Whatever you see in the type corresponds to an HTML file name inside the component directory. So if type is set to "button" then the system will get components/dropdown/button.html

The current types available for dropdown are:

  • button
  • button-split
  • button-select
  • link
Note: You can create your own version of a dropdown by creating a new file in the same directory as the component name.

options:

Options provide further customization of each type. Defualt options will be in place if options is not provided.The current options available for dropdowns are:

  • color:
    • primary
    • secondary
    • tertiary (default)
  • size:
    • large
    • default (default)
    • small
    • mini
  • orientation:
    • dropdown (default)
    • dropup

Sub menus on dropdowns

To add an extra level of dropdown menus the markup already takes care of it. All you have to do is have a different data set with nested elements.

 

sample_complex_links : {
    label : 'Sample complex links',
    items: [
        {
            id : "action_1",
            label : "Action 1",
            url : "#action"
        },
        {
            id : "action_2",
            label : "Action 2",
            url : "#action"
        },
        {
            id : "action_3",
            label : "Action 3",
            url : "#action",
            disabled: true
        },
        {
            divider :  true
        },
        {
            id : "action_4",
            label : "Action 4",
            url : "#action",
            items: [
                {
            		id : "action_4_1",
                    label : "Second Level Action 1",
                    url : "#action"
                },
                {
            		id : "action_4_2",
                    label : "Second Level Action 2",
                    url : "#action"
                },
                {
            		id : "action_4_3",
                    label : "Second Level Action 3",
                    url : "#action",
                    disabled: true
                }
            ]
        }
    ]
}

Localization of dropdown menus

In the sample data sets above you will notice the content for each menu items and button/link text. If you want all these to be localized, set the value of each labels to a textkey:


label : Sherpa.msg('textkey_label'),

Alerts

There are two category of alerts available:

  • Page level alert: This is a rendering of the alert component on the top of each page which can be called at any time with javascript.
  • General alerts: Can be inserted anywhere on a page.

Page Level Alerts

See a sample page level alert

Markup

Trigger:

<a href="#" class="btn btn-tertiary test-page-level-alert">See a sample page level alert</a>

Function to call page level alert:

$('.test-page-level-alert').click(function(event){
Sherpa.publish( "alert_message", {
      "type": "error",
      "title":"Sample Title",
      "message": "This is a sample message",
      "dismiss":true,
      "icon":"icon-alert-notice"
    });
});

General Alerts

These can be inserted inside any container.

<div data-bind="component: { 
	name :'alerts', 
	options: {
        type: 'error',
        icon:'icon-alert-notice',
        title: 'Error!',
        message: 'This is an error message.',
        dismiss:false
    }
  }">
</div>

Options

  • type:
    • success
    • information
    • attention
    • error (default)
  • icon: accepts any icon class name
  • title: Can be any text. Localized by using msg function Ex: Sherpa.msg('textkey_label')
  • message: Can be any text. Localized by using msg function Ex: Sherpa.msg('textkey_label')
  • dismiss: true (default) / false
